# Bright Precision — System Documentation

Bright Precision is the clinical assessment and AI-powered report generation platform built by **Bright Brains · Instituto da Mente**. It enables patients to complete comprehensive mental health evaluations online, upload medical documents, and receive a structured clinical report generated by AI — all reviewed and approved by a medical committee.

## 1. High-Level Architecture

```
┌─────────────────┐    ┌───────────────────────┐    ┌─────────────┐
│   Patient Form   │───▶│  Next.js API Routes   │───▶│  Supabase   │
│  (/avaliacao)    │    │  (Vercel Serverless)   │    │  (DB + Storage)
└─────────────────┘    └───────────┬───────────┘    └─────────────┘
                                   │
                        ┌──────────┴──────────┐
                        │  3-Invocation Chain  │
                        │  (Background Jobs)   │
                        └──────────┬──────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    ▼              ▼               ▼
             ┌──────────┐  ┌──────────┐   ┌──────────────┐
             │ Anthropic │  │OpenRouter│   │ n8n Webhook  │
             │ Claude    │  │(fallback)│   │ (email/notif)│
             └──────────┘  └──────────┘   └──────────────┘

┌─────────────────┐    ┌───────────────────────┐
│  Doctor Portal   │───▶│  Portal API Routes    │
│   (/portal)      │    │  (CRUD + Actions)     │
└─────────────────┘    └───────────────────────┘
```

**Stack:** Next.js 15 (App Router) on Vercel, Supabase (Postgres + Storage), Anthropic Claude Sonnet 4.6, OpenRouter fallback, jsPDF for PDF generation, n8n for notifications.

## 2. Client-Facing Assessment Flow

### 2.1 Access Gate

Patients access the assessment at `/{locale}/avaliacao`. Before seeing the form, they must enter a verification code validated against the `avaliacao_codigo` table.

- **API:** `POST /api/assessment/validate-code`
- **Storage:** Code stored in `sessionStorage` as `bb_access`
- **Dev bypass:** `NEXT_PUBLIC_AVALIACAO_DEV_MODE=true` skips the gate

### 2.2 Multi-Step Form

The assessment consists of **27 steps** (some conditional based on patient profile type). State is persisted to `localStorage` so patients can resume later.

| Step ID | Description |
|---------|-------------|
| `welcome` | Welcome and instructions |
| `dados` | Personal data (name, email, CPF, DOB, etc.) |
| `perfil` | Clinical profile selector (`adulto`, `infantil`, `neuro`, `executivo`, `longevidade`) |
| `historia` | Clinical history |
| `sintomas` | Current symptoms |
| `phq9`, `gad7`, `isi`, etc. | Standardized clinical scales (PHQ-9, GAD-7, ISI, etc.) |
| `mdq` | Mood Disorder Questionnaire |
| `medicamentos` | Current medications |
| `suplementos` | Supplements |
| `estilo` | Lifestyle factors |
| `familia` | Family history |
| `uploads` | Document uploads (PDFs, images of exams/reports) |
| `laudos` | Prior clinical reports |
| `wearables` | Wearable device data |
| `resumo` | Summary and submission |

### 2.3 File Uploads

Files are uploaded directly to Supabase Storage via signed URLs:

1. `GET /api/assessment/upload?fileName=X&fileSize=Y&contentType=Z` → returns a signed upload URL
2. Client PUTs the file directly to Supabase Storage (bucket: `assessment-pdfs`)
3. The public URL is collected and submitted with the form

### 2.4 Submission

`POST /api/assessment/submit` receives the complete `formData`, `scores`, and `uploads` array. It:

1. Inserts a row into `mental_health_evaluations`
2. Returns the `evaluationId` immediately to the client
3. Kicks off background processing via `after()` (see [Report Generation Pipeline](#4-report-generation-pipeline))

After submission, the patient sees a success screen with a link to `/portal/{evaluationId}` to track progress.

## 3. Doctor Portal (Dashboard)

### 3.1 Access

Doctors access the portal at `/{locale}/portal`. Authentication is via a portal-specific access code validated against the `assessment_access_codes` table (codes with label containing "Portal").

- **API:** `POST /api/portal/validate-code`
- **Session:** Sets an `httpOnly` cookie (`portal_session`) valid for 24 hours
- All subsequent portal API calls check for this cookie

### 3.2 Evaluation List

The main portal view shows all evaluations with filtering and sorting:

- **Filters:** status, patient profile, search (name/CPF)
- **Sort:** date, name, status
- **API:** `GET /api/portal/evaluations`

### 3.3 Evaluation Detail

Each evaluation has a detail page with two main tabs:

**Data Tab:**
- Patient demographics and form responses
- Clinical scale scores
- Uploaded documents
- Editable form fields (tracked in `form_data_history`)

**Report Tab (`ReportPreviewComponent`):**
- Live processing pipeline visualization (when generating)
- Real-time log feed from `processing_logs` with auto-scroll
- Elapsed timer based on database timestamps
- Rendered markdown report split into collapsible sections
- PDF download link
- Version history (previous report generations)

### 3.4 Doctor Actions

| Action | API Route | Description |
|--------|-----------|-------------|
| Edit form data | `PATCH /api/portal/evaluations/[id]` | Updates `form_data` and logs changes to `form_data_history` |
| Regenerate report | `POST /api/portal/evaluations/[id]/regenerate` | Re-runs the full pipeline (reuses cached extracted docs if available) |
| Approve | `POST /api/portal/evaluations/[id]/approve` | Regenerates report inline + sets `reviewer_status = approved` |
| Reject | `POST /api/portal/evaluations/[id]/reject` | Sets `reviewer_status = rejected` with `reviewer_notes` |
| Cancel job | `POST /api/portal/evaluations/[id]/cancel-job` | Cancels an in-progress generation |
| Upload documents | `POST /api/portal/evaluations/[id]/upload` | Doctors can attach additional documents |
| Delete | `DELETE /api/portal/evaluations/[id]` | Removes the evaluation |

### 3.5 Real-Time Progress UI

When a report is generating, `ReportPreviewComponent` shows a 6-step pipeline:

1. **Inicializando** — Job dispatched
2. **Extraindo documentos** — PDF/image extraction in progress (shows X of Y)
3. **Gerando relatório (IA)** — LLM generation (shows stage 1 or stage 2)
4. **Montando PDF** — PDF being built with jsPDF
5. **Salvando** — PDF uploaded to Supabase Storage
6. **Notificando** — Webhook sent to n8n

A live log feed below the pipeline shows timestamped entries from the `processing_logs` column, updated via polling.

## 4. Report Generation Pipeline

### 4.1 Overview

Report generation is a multi-step background process split across **three separate Vercel function invocations** to work within the 300-second (5 min) per-function limit on Vercel Pro.

```
Invocation 1: /api/assessment/submit (or /regenerate)
  └─ Document extraction (parallel, up to 10 concurrent)
  └─ Save extracted_documents_text to DB
  └─ HTTP POST → /api/assessment/continue-report?phase=stage1

Invocation 2: /api/assessment/continue-report (phase=stage1)
  └─ LLM call: Sections 1–5 (clinical analysis)
  └─ Save report_stage1 to DB
  └─ HTTP POST → /api/assessment/continue-report?phase=stage2

Invocation 3: /api/assessment/continue-report (phase=stage2)
  └─ LLM call: Sections 6–10 (treatment & monitoring)
  └─ Combine stage1 + stage2 → final report_markdown
  └─ Build PDF (jsPDF)
  └─ Upload PDF to Supabase Storage
  └─ Send notification via n8n webhook
  └─ Set status = completed
```

Each invocation gets its own fresh 300-second timeout budget.

### 4.2 Document Extraction

Uploaded files (PDFs, images) are processed by the AI to extract clinically relevant data:

- **PDFs:** Split into chunks of ≤90 pages per LLM call (max 32MB per chunk)
- **Images:** JPEG, PNG, GIF, WebP supported (max 20MB)
- **Concurrency:** Up to 10 parallel extraction jobs
- **Model:** `claude-sonnet-4-6` via Anthropic SDK
- **Fallback:** OpenRouter (`anthropic/claude-sonnet-4-6`) on credit exhaustion or non-retryable errors
- **Retries:** Up to 3 attempts per job with exponential backoff

The extraction prompt (`DOCUMENT_EXTRACTION_SYSTEM`) instructs the AI to extract all clinical data including lab results, neuroimaging findings, neurophysiology data, neuropsychological assessments, prescriptions, and visual descriptions of charts/graphs/tables.

Extracted text is saved to `extracted_documents_text` in the database so regenerations can reuse it.

### 4.3 Report Stages

The report is generated in two LLM calls:

**Stage 1 — Sections 1–5 (Clinical Analysis):**
- Input: Full patient form data + all extracted document text
- Sections: Executive Summary, Technical Data Integration, Diagnostic Hypotheses (CID-10), Risk Stratification, Scientific Foundation
- Max output: 12,288 tokens

**Stage 2 — Sections 6–10 (Treatment & Monitoring):**
- Input: Patient form data + Stage 1 output (NOT the raw extracted docs — reduces input size significantly)
- Sections: Therapeutic Suggestions, Monitoring Plan, References, CFM Compliance Declaration, Final Notes
- Max output: 12,288 tokens

Both stages use the CFM nº 2.454/2026 compliance preamble ensuring all outputs are framed as non-binding suggestions to the Medical Committee.

### 4.4 LLM Call Strategy

The `callLLM` function implements a robust calling strategy:

1. **Primary:** Anthropic SDK with streaming (`client.messages.stream()`)
2. **Fallback trigger:** Credit exhaustion or non-retryable errors → switch to OpenRouter
3. **OpenRouter:** Uses SSE streaming
4. **Retries:** Up to 3 attempts with exponential backoff (5s, 10s, 30s for rate limits)
5. **Progress logging:** Every 15 seconds, token count is logged to DB for UI visibility
6. **Cost tracking:** Each call logs estimated cost based on `(input×3 + output×15) / 1M`

### 4.5 PDF Generation

After the markdown report is complete, `buildPdf()` (using jsPDF) creates a branded PDF:

- Custom fonts: KMRApparat (Regular, Bold)
- Sections parsed from markdown headers
- Branded footer on each page
- Uploaded to Supabase Storage bucket `assessment-pdfs`

### 4.6 Notifications

On successful completion (submit mode), a webhook is sent to n8n:

- **Event:** `assessment_report_ready`
- **Payload:** evaluation_id, patient_name, report_url, patient_email, patient_phone, patient_profile
- **Error event:** `assessment_report_error` (sent on failure)

## 5. Vercel Serverless Architecture

### 5.1 The 300-Second Problem

Vercel Pro plan enforces a **300-second (5 min) maximum duration** per serverless function invocation. The full report pipeline (extraction + 2 LLM calls + PDF) can exceed this, so it's split into a chain of independent invocations.

### 5.2 Function Configuration

```json
// frontend/vercel.json
{
  "functions": {
    "app/api/assessment/submit/route.ts": { "maxDuration": 300 },
    "app/api/assessment/continue-report/route.ts": { "maxDuration": 300 },
    "app/api/portal/evaluations/[id]/regenerate/route.ts": { "maxDuration": 300 }
  }
}
```

### 5.3 Chaining Mechanism

Each function uses Next.js `after()` to do its heavy work after sending an HTTP response. When it finishes its phase, it makes an HTTP POST to the next function's endpoint, which starts a **new Vercel function invocation** with a fresh 300s timer.

The target URL is resolved by `getProductionUrl()`:
1. `SITE_URL` (explicit env var) — preferred
2. `VERCEL_PROJECT_PRODUCTION_URL` — auto-set by Vercel
3. `VERCEL_URL` — fallback
4. `localhost:3000` — local dev

This targets the **production domain** specifically, avoiding Vercel's SSO deployment protection on preview URLs (which would return 401).

### 5.4 Status Tracking

The `status` column in the database tracks progress with timestamped prefixes:
- `processing_dispatching_{timestamp}` — job queued
- `processing_report_{timestamp}` — extraction starting
- `processing_extract_{n}_of_{total}` — document extraction progress
- `processing_stage_1_of_2` — LLM Stage 1
- `processing_stage_2_of_2` — LLM Stage 2
- `processing_pdf` — PDF building
- `processing_upload` — PDF upload
- `processing_notify` — sending webhook
- `completed` — done
- `error` — failed (details in `processing_error`)

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid | Primary key |

| `patient_birth_date` | text | Date of birth |

| `patient_profile` | text | `adulto`, `infantil`, `neuro`, `executivo`, `longevidade` |
| `form_data` | jsonb | Complete assessment form data (~86 fields) |
| `scores` | jsonb | Computed clinical scale scores |
| `status` | text | Current processing status (see 5.4) |
| `doctor_uploads` | jsonb | Array of `{ name, url, type, path, uploaded_at }` |
| `extracted_documents_text` | text | Cached extracted text from all documents |
| `report_stage1` | text | Intermediate output from Stage 1 LLM call |
| `report_markdown` | text | Final combined markdown report |
| `report_pdf_url` | text | Public URL of generated PDF |
| `report_history` | jsonb | Array of `{ report_markdown, report_pdf_url, generated_at }` |
| `reviewer_status` | text | `pending_review`, `approved`, `rejected` |
| `reviewer_notes` | text | Doctor's review notes |
| `approved_at` | timestamptz | Approval timestamp |
| `approved_by` | text | Approver identifier |
| `form_data_history` | jsonb | Audit log of form edits |
| `processing_error` | text | Error message if generation failed |
| `processing_logs` | jsonb | Array of `{ t: timestamp_ms, m: message }` for live log feed |
| `created_at` | timestamptz | Row creation time |

### 6.2 `avaliacao_codigo` (assessment access)

| Column | Type | Description |
|--------|------|-------------|
| `codigo_verificacao` | text | Verification code for patient access |
| `ativo` | boolean | Whether the code is active |

### 6.3 `assessment_access_codes` (portal access)

| Column | Type | Description |
|--------|------|-------------|
| `code` | text | Access code |
| `label` | text | Label (must contain "Portal" for portal codes) |
| `active` | boolean | Whether the code is active |

### 6.4 RPC Functions

- **`append_processing_log(eval_id uuid, log_entry jsonb)`** — Atomically appends a log entry to the `processing_logs` array. Falls back to a direct update if RPC fails.

### 6.5 Storage

- **Bucket:** `assessment-pdfs`
- **Upload paths:** `uploads/pending/{uuid}/{timestamp}.{ext}`
- **Report PDFs:** `report_{evaluationId}_{timestamp}.pdf`

## 7. Authentication

### 7.1 Patient Assessment

- Simple code-based gate via `POST /api/assessment/validate-code`
- Code validated against `avaliacao_codigo` table
- Stored in `sessionStorage` — session-only, no persistent auth

### 7.2 Doctor Portal

- Code-based gate via `POST /api/portal/validate-code`
- Validated against `assessment_access_codes` where `label LIKE '%Portal%'`
- Creates an `httpOnly`, `secure`, `sameSite=strict` cookie (`portal_session`) with a random UUID
- All portal API routes check for this cookie — returns 401 if missing
- Cookie expires after 24 hours

### 7.3 Internal API Calls

- The chained invocations between Vercel functions use direct HTTP POST calls targeting the production URL
- No auth token is exchanged between internal calls — they rely on the production domain not having deployment protection
- Supabase calls use the `SUPABASE_SERVICE_ROLE_KEY` (server-side only)
